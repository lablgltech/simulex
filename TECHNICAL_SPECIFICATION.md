# Техническое задание на платформу Simulex

**Версия документа:** 2.0  
**Статус:** действующий  
**Объект:** веб-платформа проведения многоэтапных обучающих и имитационных сценариев («кейсов») с авторизацией участников, хранением состояния в PostgreSQL, административными функциями и формированием отчётности.

---

## 1. Введение

### 1.1. Цель документа

Зафиксировать назначение платформы Simulex, границы системы, архитектурные решения, состав программных модулей, используемые технологии, требования к конфигурации окружения и критерии приёмки результатов разработки и сопровождения.

### 1.2. Термины

| Термин | Определение |
|--------|-------------|
| **Кейс** | Логически цельный сценарий: метаданные, список этапов, действия, при необходимости ссылки на внешние ресурсы (файлы в `data/cases/`). |
| **Этап** | Единица прохождения кейса с собственным типом, конфигурацией действий и правилами перехода. |
| **Сессия** | Сохранённое состояние прохождения кейса пользователем (текущий этап, выполненные действия, ресурсы, служебные поля). |
| **Участник** | Пользователь с ролью, допускающей запуск и прохождение кейсов. |
| **Администратор / суперпользователь** | Роли с расширенным доступом к каталогу кейсов, пользователям, дашбордам и служебным операциям (по политике разграничения в приложении). |

### 1.3. Область применения

Платформа предназначена для организаций, проводящих обучение или оценку компетенций в формате пошаговых сценариев с фиксацией решений и построением отчётов. Конкретный учебный контент кейсов поставляется отдельно от ядра платформы (JSON и файлы ресурсов); в репозитории может находиться **демонстрационная конфигурация** с минимальным кейсом для проверки работоспособности цепочки «вход → сессия → этап → отчёт».

## 2. Назначение и функции системы

### 2.1. Назначение

Обеспечить:

1. **Идентификацию и авторизацию** пользователей с выдачей токена доступа (JWT) и поддержкой ролей.  
2. **Каталог кейсов** — выбор опубликованного сценария, получение полной конфигурации кейса для клиента (с учётом материализации контента на сервере).  
3. **Управление игровой сессией** — создание сессии при выборе кейса, выполнение действий на этапе, завершение этапов, переход к отчёту.  
4. **Отчётность** — генерация и отображение отчёта по завершённому прохождению (включая опциональные расширенные блоки: нормализация метрик, ИИ-нарратив при наличии ключей внешних сервисов).  
5. **Администрирование** — управление пользователями и кейсами, дашборды и вспомогательные операции (в объёме, реализованном в модуле `backend/routers/admin.py` и связанных сервисах).  
6. **Интеграцию с внешними сервисами** — при необходимости: LLM для чатов, переговоров, отчётов; опционально Redis для кэша и блокировок (переменные окружения).

### 2.2. Функциональные требования (верхний уровень)

| ID | Требование |
|----|------------|
| F-01 | Пользователь регистрируется в системе или создаётся администратором; вход по учётным данным с получением JWT. |
| F-02 | Клиент получает список кейсов со статусом «опубликован» (`GET /api/cases` или эквивалент). |
| F-03 | Клиент запрашивает полное описание кейса для игры (`GET /api/case` с идентификатором). |
| F-04 | Создание сессии привязано к пользователю и кейсу; состояние сохраняется в БД. |
| F-05 | Выполнение действий этапа изменяет состояние сессии согласно правилам сервиса действий. |
| F-06 | Завершение этапа переводит сессию на следующий этап или завершает сценарий. |
| F-07 | По завершении сценария доступна генерация и просмотр отчёта. |
| F-08 | Роли администратора и суперпользователя имеют доступ к расширенным экранам и API согласно реализации. |

### 2.3. Демонстрационная конфигурация клиента

В составе поставки может использоваться **режим упрощённого игрового интерфейса**: константа `PLATFORM_SHELL` в `src/config/platformShell.js`. При её включении скрываются элементы UX, связанные с мессенджером «Симуграм» и соответствующей боковой панелью; остальная кодовая база клиента сохраняется для полнофункциональных сборок. Это не ограничивает серверную модель данных (в том числе полей сессии, используемых в расширенных сценариях).

Для проверки базовой цепочки без развёрнутого контента кейса допускается тип этапа **`shell`** в JSON кейса, отображаемый компонентом **GenericStageView** (универсальный экран: вводный текст, список действий, завершение этапа при выполнении обязательных действий). Регистрация типа выполняется в `src/stages/StageRegistry.js`.

## 3. Архитектура системы

### 3.1. Общая модель

Архитектура **клиент — сервер**:

- **Клиент:** SPA на **React 18**, сборка **Create React App** (`react-scripts`). Обмен с сервером по **HTTPS/HTTP**, формат тела запросов и ответов — **JSON**. В режиме разработки запросы с порта фронтенда проксируются на API (настройка `proxy` в `package.json`).  
- **Сервер:** приложение **FastAPI** (ASGI), запуск **Uvicorn** (в эксплуатации возможен **Gunicorn** с Uvicorn worker — по скриптам деплоя).  
- **СУБД:** **PostgreSQL** — персистентное хранение пользователей, сессий, материализованного содержимого кейсов, отчётов и служебных сущностей согласно SQL-миграциям.  
- **Файловая система:** исходные описания кейсов `data/case*.json`; опционально каталог `data/cases/<идентификатор кейса>/` с Markdown, JSON и прочими ресурсами этапов.

### 3.2. Слои серверной части

| Слой | Содержание |
|------|------------|
| **HTTP / роутеры** | Модули в `backend/routers/`: маршруты, привязка к сервисам, коды ответов. |
| **Сервисы** | `backend/services/`: бизнес-логика сессий, этапов, действий, отчётов, кейсов, ИИ-взаимодействий и др. |
| **Доступ к данным** | Модуль `backend/db.py` и SQL-запросы в сервисах; миграции в `backend/migrations*.sql` и перечне в `run_migrations.py`. |
| **Утилиты** | `backend/utils/` — загрузка файлов кейсов, обогащение JSON контентом из Markdown и т.п. |

### 3.3. Слои клиентской части

| Слой | Содержание |
|------|------------|
| **Страницы** | `src/pages/`: вход, основной игровой экран (`GameView`), отчёты, админ-панель и др. |
| **Компоненты** | `src/components/`: HUD, модальные окна, представления этапов 1–4, отчёты, дашборд, редакторские оболочки и т.д. |
| **Реестр этапов** | `src/stages/StageRegistry.js` — сопоставление `stage.type` / `stage.id` с React-компонентом. |
| **Контекст и API** | `src/context/`, `src/api/` — авторизация, конфигурация запросов, обработка ошибок. |

### 3.4. Типовой поток взаимодействия

1. `POST /api/auth/login` — аутентификация; сохранение токена на клиенте.  
2. `GET /api/cases` — список кейсов.  
3. `GET /api/case?id=…` — конфигурация выбранного кейса.  
4. `POST /api/session/start` — создание сессии.  
5. `POST /api/action/execute` — действия на этапе (идентификаторы и контракт — по реализации роутера `actions`).  
6. `POST /api/stage/complete` — завершение этапа.  
7. `POST /api/report/generate` — формирование отчёта после финального этапа.

Точный перечень путей и параметров определяется исходным кодом роутеров в `backend/routers/`.

### 3.5. Состав HTTP-роутеров (реализация)

В проекте предусмотрены модули (имя файла → функциональная зона):

| Файл | Назначение |
|------|------------|
| `auth.py` | Вход, профиль, JWT. |
| `cases.py` | Каталог и выдача кейса. |
| `sessions.py` | Жизненный цикл сессии. |
| `actions.py` | Выполнение действий. |
| `stages.py` | Завершение этапов, вспомогательные операции этапов. |
| `reports.py` | Отчёты участника. |
| `dashboard.py` | Данные для дашборда руководителя. |
| `admin.py` | Административные операции. |
| `document.py` | Документы кейса. |
| `chat.py`, `tutor.py`, `ai_chat.py`, `stage1_ai.py` | Коммуникации и ИИ в полнофункциональных сценариях. |
| `stage4.py` | Специфика четвёртого этапа (кризис и др.). |
| `qa_bugs.py`, `dev_logs_viewer.py` | Вспомогательные и отладочные конечные точки. |
| `openai_proxy.py` | Проксирование запросов к LLM при необходимости. |

## 4. Данные и контент кейсов

### 4.1. Источники истины

- **Файлы `data/case*.json`** — описание кейсов для первичной загрузки и синхронизации с таблицей `"case"` в PostgreSQL (при включённой политике reseed на старте — переменная окружения `RESEED_CASES_ON_STARTUP`).  
- **БД** — оперативная выдача материализованного JSON (`case_content_json`) участникам и обновление при административных операциях.  
- **Каталог `data/cases/`** — внешние ресурсы (легенды `.md`, конфигурации этапов, договоры и т.д.), пути к которым указаны в JSON кейса.

### 4.2. Структура кейса (логическая)

Минимально JSON содержит объект `case` (или эквивалентную структуру после парсинга) с полями уровня:

- идентификатор, название, описание, статус, версия;  
- массив **`stages`** — этапы с полями `id`, `order`, `type`, `title`, `intro`, `actions`, при необходимости `resources`, `emails`, настройки таймеров и др.;  
- опционально: `lexic_initial`, `contract`, `crisis`, `simulator_intro` и другие блоки для расширенных сценариев.

Обогащение при загрузке на сервере выполняется функциями из `backend/utils/file_loader.py` (подстановка Markdown в этапы и документы этапа 1, лимиты времени из `game_config.json` и т.д.).

## 5. Нефункциональные требования

### 5.1. Совместимость

- **Клиент:** современные браузеры из списка `browserslist` в `package.json`.  
- **Сервер:** Python версии, совместимой с зависимостями `requirements.txt` (ориентир — 3.10+).  
- **СУБД:** PostgreSQL не ниже версии, поддерживаемой используемым драйвером и синтаксисом миграций (рекомендуется 14+).

### 5.2. Безопасность

- В режиме эксплуатации с `DEBUG=false` обязательно задание **`JWT_SECRET`** (см. `backend/main.py`, `backend/.env.example`).  
- Пароли пользователей хранятся в виде хеша (**passlib** / **bcrypt**).  
- Секреты и ключи API не включаются в репозиторий; используется `backend/.env`.  
- Распространение и использование программного обеспечения определяются файлом **LICENSE** в корне репозитория (проприетарная модель).

### 5.3. Конфигурируемость

Ключевые параметры задаются переменными окружения (файл `backend/.env`, шаблон `backend/.env.example`):

- `POSTGRES_DSN`, `PORT`, `DEBUG`, `JWT_SECRET`;  
- ключи и базовые URL для OpenAI / OpenRouter;  
- `RESEED_CASES_ON_STARTUP`, `GET_CASE_FROM_FILESYSTEM`, `REDIS_URL`, `DATA_DIR`, `LOG_LEVEL` и др. по необходимости.

На клиенте при отдельном хостинге API может использоваться `REACT_APP_API_URL`.

### 5.4. Сопровождаемость

- Версия API и приложения может отображаться через `backend/__version__.py`.  
- Миграции БД версионируются SQL-файлами и скриптом `run_migrations.py`.

## 8. Развёртывание и запуск

1. Установить зависимости: `npm install` (корень), `pip install -r backend/requirements.txt` в виртуальном окружении.  
2. Настроить `backend/.env` по образцу `.env.example`.  
3. Выполнить `python3 backend/run_migrations.py`.  
4. Создать пользователя: `python3 backend/create_superuser.py`.  
5. Запуск разработки: `npm run dev` из корня (или раздельно клиент и сервер по скриптам в `package.json`).  
6. Продакшен-сборка фронтенда: `npm run build`; выдача статики и проксирование на API — по конфигурации веб-сервера (примерные материалы в `deploy/`).

---

## 9. Критерии приёмки

1. Успешная сборка клиента без ошибок (`npm run build`).  
2. Запуск сервера приложений при корректном `POSTGRES_DSN` и соблюдении требований к `JWT_SECRET` / `DEBUG`.  
3. Применение миграций без ошибок на чистой БД.  
4. Вход пользователя и получение списка кейсов; загрузка конфигурации кейса с сервера.  
5. Прохождение демонстрационного сценария с типом этапа `shell`: выполнение обязательного действия, завершение этапа, переход к завершению сценария и возможность запроса отчёта согласно логике клиента.  
6. Отсутствие необработанных падений ядра API при типовых запросах без обязательной настройки LLM (поведение сервиса отчёта при ошибках внешнего API — по реализованной политике fallback).

## 10. Развитие системы

- Подключение полных кейсов: добавление файлов `data/case*.json` и каталогов `data/cases/…` с ресурсами.  
- Расширение клиентского UX: отключение режима `PLATFORM_SHELL` для отображения полного набора игровых панелей.  
- Доработка типов этапов через `StageRegistry` и специализированные компоненты в `src/components/`.  
- Расширение API и сервисов под новые виды действий и отчётности при сохранении обратной совместимости миграций БД.
